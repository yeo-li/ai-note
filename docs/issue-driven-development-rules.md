# 이슈 기반 개발 기본 룰

이 문서는 작업 요청을 받았을 때, 이슈 생성부터 PR까지 일관된 방식으로 처리하기 위한 기본 운영 규칙이다.

## 1. 작업 요청 수신 즉시 할 일

1. 새로운 작업 요청이 들어오면, 코드를 수정하기 전에 GitHub 이슈를 먼저 만든다.
2. 이슈 본문은 한국어로 상세하게 작성한다.
3. 이슈에는 최소 아래 내용을 포함한다.
   - 작업 배경과 목적
   - 구현 범위와 제외 범위
   - 세부 작업 항목
   - 완료 조건(acceptance criteria)
   - 선행 이슈 (`없음` 또는 `#번호`)
   - 예상 영향 파일/모듈
   - 검증 방법

## 2. 큰 작업은 반드시 분할

1. 단일 이슈가 과도하게 크다고 판단되면, 작업을 작은 태스크로 분해해 이슈를 여러 개 만든다.
2. 분할된 이슈 각각은 독립적으로 구현, 검증, 리뷰가 가능해야 한다.
3. 분할 이슈의 설명도 한국어로 구체적으로 작성한다.

## 3. 브랜치 전략

1. 모든 작업 브랜치는 `develop`에서 분기한다.
2. Codex 작업 브랜치 이름 형식은 아래를 사용한다.

```text
codex/s<번호>-<트랙>-<작업>
```

예시:
- `codex/s29-monorepo-workspaces`
- `codex/s30-server-bootstrap`
- `codex/s31-shared-contract-foundation`
3. 이슈별 worktree를 쓸 때는 활성 브랜치와 worktree를 1:1로 대응시킨다.

## 4. 이슈 기반 개발 원칙

1. 개발은 항상 이슈를 기준으로 진행한다.
2. 시작 시 이슈를 본인에게 할당(assignee)한다.
3. 하나의 브랜치에서는 하나의 이슈 범위만 처리한다.
4. 여러 이슈를 동시에 처리하지 않는다.
5. 작업 우선순위는 현재 리스크와 의존성을 기준으로 스스로 판단해 정한다.

## 4-1. 의존성 / 상태 라벨 규칙

1. 선행 작업이 필요한 이슈에는 본문에 반드시 `## 선행 이슈` 섹션을 둔다.
2. 선행 이슈가 없으면 `없음`이라고 명시한다.
3. 상태 라벨은 아래 3개만 사용한다.
   - `status:blocked`: 선행 이슈가 닫히기 전이라 시작하면 안 되는 상태
   - `status:ready`: 선행 조건이 충족되어 바로 진행 가능한 상태
   - `status:in-progress`: 현재 작업 중인 상태
4. 선행 이슈가 하나라도 열려 있으면 후속 이슈를 `status:blocked`로 유지한다.
5. 선행 이슈가 모두 닫히면 후속 이슈를 `status:ready`로 바꾼다.
6. 실제 작업을 시작하면 `status:in-progress`로 바꾼다.

## 5. 커밋 규칙

1. 커밋은 기능 단위로 나누어 의미 있게 작성한다.
2. 커밋 메시지는 한국어 중심으로 작성한다. (기술 키워드 수준의 영어는 허용)
3. 이슈 번호를 커밋 메시지에 포함해 추적 가능하게 유지한다.

예시:
- `[#123] 메모 검색 결과 정렬 기준을 최신 수정순으로 변경`
- `[#124] 빌드 스크립트 중복 옵션 제거`

## 6. PR 규칙

1. 이슈 작업 완료 후 관련 커밋을 푸시하고 PR을 생성한다.
2. PR의 base 브랜치는 항상 `develop`으로 고정한다.
3. PR의 compare(head)는 해당 이슈 작업 브랜치로 설정한다.
4. PR 제목과 본문은 한국어 중심으로 작성한다. (필요한 영어 용어만 제한적으로 사용)
5. PR 본문에는 최소 아래를 포함한다.
   - 무엇을 변경했는지
   - 왜 변경했는지
   - 변경 파일/영향 범위
   - 검증 방법과 결과
   - `Depends-On` (`없음` 또는 선행 이슈 `#번호`)
   - 연결 이슈 (`Closes #이슈번호`)
6. `develop` 대상 PR은 `PR Dependency Check`를 통과해야만 병합한다.
7. `PR Dependency Check`는 아래 순서로 의존성을 검사한다.
   - PR 본문의 `## Depends-On` 섹션
   - PR 본문의 `Closes #...`로 연결된 이슈의 `## 선행 이슈` 섹션
8. GitHub 브랜치 보호 규칙에서는 workflow `PR Dependency Check`의 job `dependency-check`를 required status check로 수동 등록한다.

## 7. 최소 검증 기준

1. PR 생성 전 최소 `npm run build:renderer`를 통과해야 한다.
2. 필요 시 기능별 추가 검증(테스트, 수동 시나리오)을 PR 본문에 기록한다.

## 8. 작성 언어 원칙

1. 이슈/PR/커밋 설명은 모두 자연스러운 한국어로 작성한다.
2. 전체 문장을 영어로만 작성하지 않는다.
3. 개발 문맥상 필요한 용어(브랜치명, 명령어, 코드 식별자)는 영어를 그대로 사용한다.
