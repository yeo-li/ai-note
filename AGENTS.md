# AGENTS: ai-note (Codex Working Rules)

이 문서는 `/Users/yeoli/git/ai-note`에서 Codex가 매 작업 시작 전에 반드시 읽고 따라야 할 규칙이다.

## 1) 브랜치/PR 원칙 (main 직접 작업 금지)

- `main`에는 직접 커밋하지 않는다.
- 작업은 이슈 단위 브랜치(`codex/s<번호>-<트랙>-<작업>`)로 분리한다.
- 한 브랜치에서 한 이슈 범위만 처리한다.
- PR에는 목적/범위/변경 파일/테스트(검증)만 간단히 기록한다.
- 브랜치 머지 전 필수 검증: 최소 `npm run build:renderer` 통과.

## 2) UI 작업 주의점 (Desktop + Electron + macOS)

- Electron 윈도우는 창 스타일(특히 `hiddenInset`)을 고려해 레이아웃에 상단 안전영역을 둔다.
- macOS의 제목 표시줄 버튼/교차오버 UI가 콘텐츠와 겹치지 않도록 `padding-top` 또는 전용 클래스에서 오프셋을 적용한다.
- 화면 전체를 사용해야 하는 모드에서는 바깥 테두리·드래그존·불필요한 프레임 chrome을 제거한다.
- 리스트/에디터 분할 레이아웃은 창 크기 변화에 따라 깨지지 않게 반응형으로 구현한다.

## 3) 메모 앱 핵심 동작 원칙

- 검색/선택은 분리한다.
  - 검색 결과가 바뀌어도 현재 선택이 결과 집합에 있으면 유지한다.
  - 검색 결과 0건이면 에디터는 빈/미검색 상태로 전환해, 이전 메모 오탐 편집을 막는다.
- destructive action은 반드시 `confirm` 또는 `undo` 경로를 거친다.
  - 삭제는 즉시 반영하지 않고 확인 또는 1회 undo 보호를 기본값으로 둔다.
  - 마지막 항목 삭제 시 자동 새 항목 강제 생성은 금지한다.
- AI 텍스트 변환(요약/문체 변환)은 preview-before-apply 원칙.
  - 즉시 덮어쓰기 금지, 원문 보존/복원 경로를 항상 제공한다.

## 4) 작업 체크리스트

### 작업 시작 전
- 관련 이슈 목표, 적용 범위, 영향 파일을 1줄로 정리한다.
- 동일 화면/컴포넌트의 기존 패턴(선택-검색-버튼 상태)을 확인한다.
- macOS hiddenInset 여부를 확인하고 필요한 오프셋 적용 여부를 결정한다.

### 작업 후
- 코드가 규칙(브랜치/PR, destructive 보호, 검색-선택 분리, AI preview)과 맞는지 검토한다.
- 최소 검증 결과를 기록한다 (`npm run build:renderer`).
- 결과/한계/추가 확인 포인트를 PR 본문 또는 댓글에 남긴다.
